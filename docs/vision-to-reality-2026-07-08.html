The six build areas

Each card: the recommendation, why, the pitfall, and the primary source. Chips mark status — proven (already working), ready (drop-in), nuance (read the caveat).

1 · Terminal render — Canvas now, WebGL if IPC-bound later nuance
xterm.js's own benchmarks show a WebGL cell-grid renderer is 3–10× faster than Canvas 2D (300×80 on macOS: 15.3ms → 3.7ms/frame), because it batches every cell into one Float32Array uploaded to a GPU shader instead of per-character CPU draws.
but hereThat benchmark is for xterm.js, where JS does the VT parsing. In this app parsing is already done in Rust (libghostty-vt) and only the cell grid crosses to the webview — so the real bottleneck is IPC volume, not draw calls. Canvas 2D already passed the render test. WebGL is the scaling fallback if fast-output stutters, not a day-one rewrite.
pitfallDon't cargo-cult "WebGL is faster." First fix IPC: ship dirty-region deltas, not full-grid snapshots (Ghostty's own render thread does exactly this — damage-tracks cells, rebuilds only what changed).
src xtermjs/xterm.js#1790 · Ghostty rendering
2 · Editor — @uiw/react-codemirror (CodeMirror 6) ready
Embed CM6 through the maintained @uiw/react-codemirror v4.25.x binding (deps pinned to CM6 6.x) — it supplies the React lifecycle glue so you don't hand-roll the imperative↔declarative bridge. Add languages via the extensions prop (@codemirror/lang-javascript, lang-rust, lang-python); StreamLanguage + legacy-modes as fallback.
vs monacoCM6 core ~300KB vs Monaco ~5–10MB (17–33× larger). Monaco doesn't support webview apps; CM6 does (ships in Safari/Chrome/Brackets). For a lightweight desktop editor CM6 is the clear pick — matches the locked stack.
pitfallRaw CM6 has no React integration — using it directly forces a manual useCodeMirror bridge. Use the @uiw wrapper, don't reinvent it.
src uiwjs/react-codemirror · size comparison
3 · File rail — ignore + notify + React Arborist ready
Three drop-ins from the ripgrep ecosystem + one React lib cover the whole rail:
walkignore::WalkBuilder — respects .gitignore, .ignore, .git/info/exclude and global gitignore by default. Gitignore-aware filtering for free.
watchnotify (FSEvents on macOS; used by rust-analyzer/cargo-watch) + notify-debouncer-mini — debouncing is NOT in notify core; the debouncer emits one event per timeframe per file.
renderReact Arborist 3.12.0 — virtualized tree explicitly targeting a VS Code sidebar; ships inline rename, keyboard nav, ARIA, filtering, drag-sort.
pitfallWatching a repo without the debouncer = FSEvents event storms on every npm install / build. Wire the debouncer from day one.
src ignore · notify · react-arborist
4 · Persistence — Store plugin now, SQL plugin later ready
Tauri Store plugin (persistent KV → JSON file, autosaves 100ms after change) covers v0/v0.5 workspace state: last folder, editor tabs, pane layout. Move to the SQL plugin (SQLite via --features sqlite, transactional migrations) only when state turns relational (project × pane × process history in v1).
ruleStore for simple serializable state; SQL when you need queries/relations. Don't reach for SQLite before the data is actually relational — matches hashmark's proven pattern.
pitfallStore's autosave is debounced — a hard crash can lose the last <100ms. Fine for layout; call save() explicitly after meaningful edits if it ever matters.
src Store plugin · SQL plugin
5 · Multi-agent processes — portable-pty (already in use) proven
The pty layer SPIKE-2 already uses scales to many panes: native_pty_system() → openpty() → slave.spawn_command(); try_clone_reader() / take_writer() drive a per-pane loop; Child / ChildKiller / ExitStatus + MasterPty::resize() handle lifecycle. WezTerm runs the same crate this way per pane.
patternSPIKE-2's two-thread-per-pty design (blocking reader thread → channel → terminal thread owning the !Send engine) is exactly WezTerm's LocalPane shape. Already correct — reuse it per pane.
openMemory/CPU ceiling at 4–8+ concurrent agents, idle-suspension, and zombie reaping are not yet verified — measure at v1 (see open questions).
src portable-pty · WezTerm pty mgmt
6 · Packaging — ad-hoc sign for local; notarize only to share ready
For your own machine, ad-hoc signing is enough (on Apple Silicon you whitelist it once in Privacy & Security). Only distributing to others needs a Developer ID cert + notarization ($99/yr Apple account). Tauri's CLI builds .app / .dmg directly.
pitfallAdding a externalBin sidecar breaks notarization in Tauri 2.x (invalid-signature error). If you ever bundle a helper binary, budget time for the signing dance — or avoid sidecars.
src Tauri macOS signing · sidecar notarize bug

One phantom constraint — killed

Refuted 0–3

"Safari/WKWebView lacks WebGL2, so a GPU terminal renderer won't work in a Tauri macOS webview." False. WebGL2 works in WebKit. If Canvas ever becomes the bottleneck, the GPU path is genuinely available — don't design around a constraint that doesn't exist.

What research did NOT settle

Honest gaps — these need measurement in the build, not another search. Named so they don't masquerade as covered.

Q1The exact IPC pattern for high-frequency pty output — batched binary frames vs channel vs shared memory — and the WKWebView bridge's throughput ceiling during a TUI full-repaint. Resolve by measuring v0/v1 fast-output.
Q2Whether CM6 truly beats Monaco for >1MB files here (bundle size favors CM6; large-file behavior unproven for this app).
Q3Real memory/CPU ceilings and backpressure/idle-suspension patterns at 4–8+ concurrent agent ptys; whether portable-pty needs extra zombie-reaping on macOS. Measure at v1.
Q4How much of the xterm.js WebGL win transfers given VT parsing is already in Rust and only the grid ships to the webview (the bottleneck likely moves from draw to IPC).
