Keelhouse - User Flow Report

2026-07-08 · Workbench shell, file navigation, editor save, and error recovery guidance for the active editor parity roadmap slice.

userflow flow-app-shell flow-navigation flow-errors

Summary

The proven shell for Keelhouse is a desktop productivity sidebar-plus-main layout with independent file rail, editor, and terminal panes. The current implementation has crossed the basic editor threshold; the next parity work should preserve pane orientation, restore editor position per file, protect unsaved work, and keep every navigation/action reachable by pointer plus keyboard. The biggest flow risk is not visual polish; it is losing work or focus when switching files/projects while agents are running.

Flow: Open, Edit, Save a File

Pattern source: flow-app-shell + flow-navigation + flow-errors · Entry point: selected project workspace

Steps

  1. 1Project workbench loaded

    The app paints stable chrome immediately: file rail on the left, editor in the main pane, and terminal below. The workspace name anchors the rail and the editor empty state says what to do next.

    Primary action: Select a file -> file opens in the editor pane.

    If no workspace is loaded -> use Open Folder, then restore the rail/editor/terminal shell.
  2. 2File selected from rail

    The file row becomes active, the editor tab title updates, a path bar shows the absolute file path, and CodeMirror displays line numbers plus syntax highlighting where supported.

    Primary action: Edit text -> dirty state appears in the editor header.

    If the file is too large, binary, outside the workspace, or unreadable -> preserve current editor state and show a persistent editor error with the cause and next action.
  3. 3Unsaved edit in progress

    The editor remains the focused work surface. Dirty state must be visible in the tab/header and should later be reflected in the file rail and tab list.

    Primary action: Save -> write the file and return to saved state.

    If the user switches files/projects with unsaved edits -> preserve the draft, block destructive replacement, or ask with a consequence-naming dialog.
  4. 4Saved file

    The editor returns to saved state, the file remains selected, and the user's cursor/scroll position should stay stable. The terminal pane should not steal focus during save.

    Primary action: Continue editing or switch context -> same shell, current orientation preserved.

    If save fails -> keep edits in the editor, show the failure at the editor action area, and offer Retry without clearing content.

Flow: Editor Parity UX Slice

Pattern source: flow-app-shell + flow-navigation · Entry point: verified EDITOR roadmap item

Steps

  1. 1Reopen a previously viewed file

    Opening a file should restore its scroll position, cursor, and selection. This matches the "each pane preserves state" expectation from desktop productivity shells.

    Primary action: Open file -> editor restores view state.

  2. 2Switch active file from rail

    The rail should reveal and highlight the active file, while the editor title and path bar update to the same object. Users must be able to answer "where am I?" from the chrome.

    Primary action: Select another file -> previous file state is saved locally, new file opens.

    If the current file is dirty -> preserve draft and require a clear save/discard choice before replacing the buffer.
  3. 3Resize the app

    The shell should stay usable at small desktop widths: file rail remains useful, editor keeps a minimum readable width, and terminal does not squeeze the editor into a dead zone.

    Primary action: Resize window -> pane constraints preserve the active work surface.

  4. 4Screenshot QA

    Capture desktop-sized proof for selected-file, dirty-file, save-error, and narrow-width states. Appearance is part of the contract for editor parity.

    Primary action: Record QA artifact -> roadmap item can be verified without relying on memory.

Flow: Save Failure and Recovery

Pattern source: flow-errors · Entry point: Save button or Cmd+S

Steps

  1. 1Save requested

    The editor disables duplicate save actions and attempts the workspace-scoped write.

    Primary action: Save -> saved state or scoped error.

  2. 2Save fails

    The text remains in the editor. The error appears in the editor region, states the human cause, and keeps the user in the same context.

    Primary action: Retry save -> same buffer writes again.

    If the file changed externally -> show both versions or a reload/save-as decision, never silent overwrite.
  3. 3Recovery completes

    The editor returns to saved state and confirms reconciliation through status text, not a transient-only toast.

    Primary action: Continue -> resume editing with the same cursor/scroll context.

Open questions & parked items

  • Should dirty-file switch protection block immediately, or preserve dirty buffers in a per-file draft map and allow switching?
  • Should editor view state persist only during the session first, then across relaunch in the later session-restore work?
  • External conflict handling belongs to SAVE-CONFLICTS, but EDITOR-PARITY-UX should avoid introducing silent overwrite paths.
  • The shell should be screenshot-tested at a normal desktop size and at 1024x640 before marking parity done.
